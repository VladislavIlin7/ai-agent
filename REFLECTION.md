# Reflection

## 1. Почему эта задача

Эта задача показалась мне хорошей для хакатона, потому что она находится на стыке реальной рутины и понятного агентного поведения. Почта до сих пор остается местом, куда приходят интервью, созвоны, дедлайны, приглашения на вебинары, учебные события и рабочие договоренности. При этом человек часто видит письмо, думает "надо добавить в календарь", отвлекается и забывает. Проблема маленькая, но болезненная: пропущенный дедлайн или забытое интервью стоит дороже, чем кажется.

Мне было важно сделать не просто чат-бота, который отвечает текстом, а агента, который реально вызывает tools. В этом проекте есть явная цепочка действий: прочитать ограниченное число писем, отобрать кандидатов, извлечь структуру события, показать preview и создать запись в календаре. Такой сценарий хорошо демонстрирует отличие агента от обычного LLM wrapper: модель не является всем приложением, она отвечает только за неоднозначную часть — понимание письма. Gmail и Calendar вынесены в отдельные инструменты, которые можно тестировать, заменять и дорабатывать независимо.

CLI-формат выбран осознанно. Для MVP он быстрее веб-интерфейса, проще в запуске и честнее показывает, что происходит внутри. На хакатоне это полезно: можно открыть терминал и видеть, какие письма обработаны, где сработал fallback, какие данные уйдут в календарь. Demo mode тоже важен: проект можно показать одной командой без OAuth, без реальных писем и без риска создать лишние события.

## 2. Где агент сломался

Первый failure case — неоднозначные даты и время. Письма часто содержат фразы вроде "next Friday", "tomorrow afternoon", "after lunch", "по Москве" или "let's do the same slot". Для человека это понятно из контекста, но для MVP контекст ограничен одним письмом. Если письмо пришло из другой таймзоны или продолжает предыдущую переписку, модель может выбрать неправильную дату. В текущей версии нет отдельного temporal parser, нет учета даты получения письма как обязательного контекста и нет проверки результата с пользователем в виде редактируемой формы.

Второй failure case — обновления событий. Реальные письма часто не создают новое событие, а меняют старое: "переносим на 15:00", "ссылка обновилась", "созвон отменяется". MVP пока не хранит состояние, не связывает Gmail message id с Calendar event id и не умеет отличать новое событие от изменения существующего. Из-за этого повторный запуск может создать дубликат, а письмо с отменой может быть ошибочно превращено в новое событие.

Третий риск — лимиты и нестабильность LLM endpoint. Во время подключения Qwen-compatible endpoint появились rate limit ошибки. Код не должен падать в таком случае, поэтому есть fallback extractor, но качество fallback ниже: он подходит для demo и простых дат, но не заменяет модель. Это честный компромисс MVP: приложение остается работоспособным, но результат может быть менее точным.

## 3. Что сделал бы дальше

Первым делом я бы добавил JSON Schema validation и retry-loop для LLM. Сейчас extractor ожидает строгий JSON, но production-версия должна явно валидировать поля, формат даты, формат времени и допустимые timezone. При ошибке нужно отправлять модели короткий repair prompt, а не сразу уходить в fallback.

Следующий шаг — дедупликация и состояние. Я бы добавил SQLite базу с таблицей `processed_messages`: Gmail message id, hash extracted event, Calendar event id, время обработки и статус. Это позволило бы безопасно запускать агента несколько раз, обновлять уже созданные события и не плодить дубли.

Еще одно важное улучшение — пакетное подтверждение. Вместо вопроса перед каждым событием агент мог бы вывести таблицу найденных событий: title, date, time, source subject, confidence. Пользователь выбирает, что создать, а что пропустить. Это снизит риск неправильных событий и сделает UX лучше даже в CLI.

Также я бы добавил поддержку нескольких событий в одном письме, обработку cancel/update сценариев, выбор `--calendar-id`, `--timezone`, `--dry-run`, Gmail query фильтр и тесты с моками Google API. Для безопасности стоит явнее показывать, какой фрагмент письма отправляется в LLM, и добавить настройку максимальной длины текста. Так MVP можно превратить в надежный ежедневный инструмент, а не только в демонстрацию идеи.
